# GroundGuard 项目计划书

## 1. 目标

做一个能积累真实 star 数、体现个人工程能力的开源项目，选题必须满足两个条件：

1. 贴合自己实际做过、验证过的经验，不是凭空想的功能列表。
2. 在当前 AI 基础设施赛道里能找到一个没被巨头占位、也没被做成小众学术工具的空档。

## 2. 为什么不做"通用 AI Agent 观测/评测平台"

最初的想法（内部代号 AgentOps Lite）是做一个"轻量本地优先的 Agent 观测 + CI 质量门禁"工具。调研后发现这个赛道已经严重饱和，成熟开源项目和大厂工具正在快速占位：

| 方向 | 代表项目 | 现状 |
| --- | --- | --- |
| Agent 观测/监控 | AgentOps-AI/agentops | 约 5.7k star，MIT，主流框架集成覆盖广 |
| LLM tracing / evals | Langfuse | 约 30k star，已覆盖观测、prompt 管理、evals、dataset、playground |
| LLM tracing | Arize Phoenix | OpenTelemetry 标准，开源自托管首选 |
| 轻量代理式观测 | Helicone | 改一行 base_url 即可接入 |
| CI 内 YAML 评测门禁 | promptfoo | 约 23k star，已成为 OpenAI 的一部分，继续 MIT 开源 |
| 通用记忆层 | Mem0 | 约 60k star，Apache-2.0，"AI Agent 记忆层"定位已经很强 |
| 小模型分流路由 | RouteLLM / vLLM Semantic Router | 分别有学术机构和红帽背书 |
| 通用幻觉检测 | UQLM、Loki、RefChecker | 学术气重，star 天花板普遍在 1k 以内 |

结论：这几类方向要么已经有成熟项目占住生态位（观测、记忆、CI 评测），要么是学术向的小众工具、本身 star 天花板就不高（幻觉检测）。用一个人的精力去做"更轻量版本"，功能上很难打过对方，分发资源也差一个数量级。

同时命名也有硬伤："AgentOps" 这个名字已经被 AgentOps-AI 和 Azure/agentops 两个项目占用，继续用会被搜索引擎和用户误认成"某个官方项目的精简版"。

## 3. 真正的空档：生成前的事实强约束

调研同时发现一个具体的、没被覆盖的空档：

- 现有的幻觉检测工具（UQLM、RefChecker 等）做的是**生成之后**用统计/黑盒方法给答案打分，属于事后检测，星数天花板本身就低，因为定位偏研究工具。
- 现有的观测/评测工具关注的是"记录发生了什么"和"主观质量打分"，都不对生成过程做**前置约束**。
- 没有一个开源项目把"工具调用结果"和"模型最终输出"之间的关系，做成一个可执行、可测试的强制引用机制——即模型只能引用已核实的事实，引用不到就必须显式承认不确定，而不是自由发挥。

这个机制来自真实生产场景里反复验证过的教训：工具调用型 Agent 最容易在"看起来正常"的地方出问题——明明有数据却说没有、明明没数据却编一个像样的答案。这两类问题在实际项目里都真实出现过，也已经用类似的"账本+核对"思路解决过，是有真实工程验证支撑的方向，不是凭空设想。

## 4. 定位与差异化

GroundGuard 不做以下事情，明确避免与巨头正面竞争：

- 不做通用 trace 存储和可视化 dashboard（Langfuse/Phoenix 已经做得很重很好）
- 不做主观质量打分的 LLM-as-judge 评测引擎（promptfoo/DeepEval 已经是事实标准）
- 不做通用记忆层（Mem0 已经垄断）
- 不做通用模型路由（RouteLLM/vLLM Semantic Router 已有巨头背书）

只做一件事：**给工具调用型 Agent 的最终输出放行前加一道可测试的事实闸门**。做成一个框架无关、几行代码可接入的小库，而不是一个需要理解一整套流水线概念才能用的重框架。

对外沟通口径上，避免把这个项目包装成"又一个幻觉检测器"（听起来学术、star 天花板低），而要包装成一个具体、任何做工具调用型 Agent 的人都会遇到的场景："模型明明拿到了数据却说没拿到"或"编了一个像样的假数字"——这个坑足够具体、足够普遍，容易做出一眼看懂价值的 demo。

## 5. 冷启动与增长策略

参考近期传播速度快的开源 AI 项目，共同点不是功能完整度，而是"一个任何人都能秒懂价值的具体场景 + 极低的接入门槛"。据此确定的策略：

1. **v1 只做最小闭环**：Ledger 记录 + 输出声明核对 + 必需事实覆盖检查 + Coverage Report，不做 dashboard、不做多框架适配，先把"防编造/防漏用工具事实"这两件事做扎实。
2. **用一个真实、脱敏后的失败案例做 demo**：工具明明返回了正确数据，模型却在报告里说"未能核实"或没有使用该数据——这种案例任何看过 Agent 输出的人都秒懂，比抽象的"幻觉检测"叙事更有传播力。
3. **接入门槛对标 Mem0/RouteLLM**：几行代码、一个 context manager 就能用，不要求理解账本/对账单以外的任何概念。
4. **不重复造轮子**：CI 集成阶段直接输出与 promptfoo/DeepEval 兼容的断言格式，定位成"这些工具的补充组件"，借力而非对抗，降低被现有生态排斥的风险。
5. **发布节奏**：先在小范围（个人技术圈、相关 subreddit、HN）发布可运行的 demo repo + 一段清晰的问题说明，验证是否有人产生共鸣，再决定要不要投入框架适配和可视化。

## 6. 风险与待办

- **命名**：已确认 GroundGuard 无重名冲突，仓库已在 GitHub 完成改名，本地目录已同步为 `GroundGuard`。
- **内容脱敏**：公开仓库中的所有示例、案例说明都需要用通用化、原创的表述，不能出现任何公司内部项目名称、业务代码或专有实现细节。
- **范围克制**：最大的失败模式是把 v1 做成一个大而全的框架，重蹈"AgentOps Lite"想两头都做的覆辙。任何新功能想法先问一句"这是不是巨头已经做得很好的东西"。
- **叙事与实现一致**：对外不要承诺 token 级"生成前控制"。v1 的真实能力是"输出放行前的确定性事实门禁"。
- **概念边界**：区分 `Required Fact`（本轮必须覆盖的事实）和 `Output Claim`（模型最终写出的待核实声明）。如果混用 `Claim`，后续实现和文档会互相打架。
- **下一步**：先进入 S0-S2 的受控开发：仓库骨架、数据模型、Ledger、显式事实登记、`tool_call`。在 Output Claim Extractor / Matcher / Policy 开工前，先把 fact key 引用和必需事实覆盖检查的接口定死。
